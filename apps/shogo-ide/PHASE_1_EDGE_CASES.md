# Phase 1 Edge-Case Checklist

This checklist is part of the Phase 1 Code - OSS spike. Every item must be considered before the spike graduates into desktop integration.

## Repository and upstream strategy

- [ ] Do not vendor the full Code - OSS tree into this repo during Phase 1.
- [ ] Decide whether long-term upstream lives as a separate fork repo, Git submodule, Git subtree, or scripted local checkout.
- [ ] Keep the Shogo patch set shallow and documented under `apps/shogo-ide/patches/`.
- [ ] Avoid editing VS Code workbench internals unless an extension contribution cannot solve the product requirement.
- [ ] Keep a repeatable rebase strategy against upstream `microsoft/vscode`.

## Product identity

- [ ] Replace Microsoft product names, icons, protocol names, bundle identifiers, data folder names, issue links, and documentation links.
- [ ] Use `shogo-ide://` for IDE-specific deep links and avoid colliding with the existing desktop `shogo://` app protocol until integration is explicit.
- [ ] Confirm macOS bundle ID, Windows app user model ID, Linux desktop file name, and app icon names.
- [ ] Confirm user data location does not overwrite existing `~/Library/Application Support/Shogo` desktop data.

## Licensing and marketplace

- [ ] Code - OSS is MIT, but the existing monorepo is AGPL-3.0-or-later at root; distribution boundaries need legal review.
- [ ] Do not ship Microsoft branding.
- [ ] Do not assume Microsoft Marketplace access.
- [ ] Default extension gallery should be Open VSX or a Shogo-controlled Open VSX proxy.
- [ ] Document extensions that are incompatible because they are licensed only for official Microsoft Visual Studio Code builds.

## Telemetry and privacy

- [ ] Disable Microsoft telemetry endpoints.
- [ ] Route any future Shogo telemetry through an explicit Shogo privacy policy and setting.
- [ ] Ensure crash reports do not include file contents, secrets, prompts, terminal output, or git diffs by default.
- [ ] Redact tokens from logs before attaching diagnostics to bug reports.

## Workspace trust and command safety

- [ ] Shogo edit/apply commands require workspace trust.
- [ ] Shell commands require workspace trust and explicit approval for risky commands.
- [ ] Git writes, package installs, migrations, deployments, and network writes require explicit confirmation.
- [ ] Read-only explain/summarize actions can run in restricted mode with selected context only.
- [ ] Respect `.gitignore` and support `.shogoignore` before sending context to models.

## Extension host split

- [ ] UI features should run close to the UI extension host.
- [ ] Workspace/file/indexing/terminal actions should run in the workspace extension host or local agent service.
- [ ] Web and remote workspace behavior must not assume `file://` URIs.
- [ ] Use `vscode.workspace.fs` instead of direct Node FS where possible.

## Agent service isolation

- [ ] Heavy indexing, patch application, command execution, and long-running tasks should run outside the VS Code extension host.
- [ ] Extension must reconnect if the agent service restarts.
- [ ] Tasks must persist enough state to recover or clearly fail.
- [ ] Every multi-file edit should create a checkpoint first.

## Webview security

- [ ] Use strict CSP with nonces.
- [ ] Sanitize markdown/model output before rendering.
- [ ] Validate every webview message schema.
- [ ] Restrict `localResourceRoots`.
- [ ] Do not execute command URIs generated by model output.

## Filesystem and large repo behavior

- [ ] Skip `node_modules`, `.git`, `dist`, build outputs, vendored SDKs, binaries, and oversized files during indexing.
- [ ] Support binary file detection.
- [ ] Support nested repos, worktrees, submodules, and monorepos.
- [ ] Degrade gracefully when a workspace is virtual or remote.

## Git edge cases

- [ ] Detached HEAD.
- [ ] Merge/rebase/cherry-pick in progress.
- [ ] Conflicted files and multi-stage index entries.
- [ ] Untracked or ignored generated files.
- [ ] Large diffs and binary diffs.
- [ ] Credential prompts and auth failures.
- [ ] Protected branches and force-push prevention.

## Packaging and update edge cases

- [ ] macOS signing and notarization.
- [ ] Windows signing and Squirrel/installer behavior.
- [ ] Linux AppImage/deb/rpm naming and sandbox assumptions.
- [ ] Auto-update channels: stable, beta, insiders.
- [ ] Rollback if bundled Shogo extension version is incompatible with app version.

## Current Shogo Desktop coexistence

- [ ] Phase 1 must not change the current `apps/desktop` launch path.
- [ ] Existing Monaco IDE remains the fallback while the Code - OSS path is not production-ready.
- [ ] Future integration should launch Shogo IDE from the current IDE icon without removing the old tab first.
- [ ] Data folders, auth state, local runtimes, ports, and app protocols must not conflict between current Desktop and Shogo IDE.

## Phase 1 exit checks

- [ ] `bun run --cwd apps/shogo-ide phase1:check` passes.
- [ ] Root `bun run shogo-ide:phase1:check` passes.
- [ ] Working tree contains no accidental Code - OSS source vendoring.
- [ ] No commits or pushes were made by the agent.
