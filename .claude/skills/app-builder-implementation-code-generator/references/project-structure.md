# Project Structure Patterns

## Standard Python Project Layout (kpmg-contract-parser Pattern)

**Round 2 Update**: Added completeness documentation files (TODO.md, ARCHITECTURE_DECISIONS.md, README Known Limitations)

```
.schemas/app-builder-implementation-code-generator/workspaces/{session-name}/
├── .venv/                          # Python virtual environment
├── schema.json                     # Layer 2 application schema (copied)
├── scripts/
│   └── build_types.py              # Schema → Pydantic generator
├── generated/
│   ├── __init__.py
│   └── models.py                   # Auto-generated Pydantic models
├── src/
│   └── modules/
│       ├── {module_name}/
│       │   ├── __init__.py
│       │   └── {module_name}.py   # Function stubs
│       └── ...
├── tests/
│   ├── test_{module_name}.py      # pytest tests
│   └── ...
├── requirements.txt                # Python dependencies
├── .gitignore                      # Python-specific ignores
├── README.md                       # TDD workflow guide + Known Limitations
├── TODO.md                         # NEW (Round 2) - Implementation backlog
└── ARCHITECTURE_DECISIONS.md      # NEW (Round 2) - Design rationale
```

## Directory Purposes

### Root Level

**`.venv/`** - Python virtual environment
- Created via: `python3 -m venv .venv`
- Activated: `source .venv/bin/activate`
- Contains: All Python dependencies in isolation

**`schema.json`** - SOURCE OF TRUTH
- Copied from Layer 2 workspace
- Used by `build_types.py` to generate Pydantic models
- When schema evolves, regenerate models and fix type errors

**`README.md`** - Project documentation
- Setup instructions
- TDD workflow (RED → GREEN → REFACTOR)
- Module overview table
- Traceability guide

**`requirements.txt`** - Python dependencies
- Base: pydantic, pytest, mypy, datamodel-code-generator
- Extras: Extracted from ModuleSpecification.details.libraries
- Install: `.venv/bin/pip install -r requirements.txt`

**`.gitignore`** - Version control ignores
- Python: `__pycache__/`, `*.pyc`, `.venv/`
- Testing: `.pytest_cache/`, `.coverage`
- Generated: `generated/models.py` (debatable - we commit it for drift detection)

### `scripts/` - Code Generation Tools

**`build_types.py`** - Schema → Pydantic generator
- Copied from `assets/templates/build_types.py`
- Runs datamodel-codegen
- Adds header with schema hash
- Creates `__init__.py` for generated package
- **Usage**: `python scripts/build_types.py`

**Future Scripts**:
- `validate_all.py` - Run all validation steps (syntax, types, tests)
- `run_tdd.py` - Interactive TDD helper (pick function, run tests, iterate)

### `generated/` - Auto-Generated Types

**`models.py`** - Pydantic v2 models
- ⚠️ **DO NOT EDIT MANUALLY** - will be overwritten
- Generated by `build_types.py` from `schema.json`
- Header contains: schema hash, generation timestamp, entity count
- Committed to git for drift detection (schema changes show as model diffs)

**`__init__.py`** - Package exports
- Re-exports all models: `from .models import *`
- Defines `__all__` for clean imports
- **Usage**: `from generated.models import Template, Section`

### `src/modules/` - Implementation Code

**Organization**: One directory per ModuleSpecification

```
src/modules/
├── document_parser/
│   ├── __init__.py          # Module docstring
│   └── parser.py            # All stubs for this module
├── comparison_engine/
│   ├── __init__.py
│   └── comparator.py
└── ...
```

**Module File** (`{module_name}.py`):
- Contains all function stubs from InterfaceContracts
- Functions have NotImplementedError bodies (Session 1)
- Type hints reference `generated.models` entities
- Traceability comments: `# Implements req-001`

**Module Init** (`__init__.py`):
```python
"""
{module_name} Module

Purpose: {ModuleSpecification.purpose}
Category: {ModuleSpecification.category}

Implements Requirements: {implementsRequirements[]}
"""
```

### `tests/` - Test Scaffolding

**Organization**: One test file per module

```
tests/
├── test_document_parser.py   # All tests for document_parser module
├── test_comparison_engine.py
└── ...
```

**Test File** (`test_{module_name}.py`):
- Contains all test functions from TestSpecifications
- Tests have placeholders (Session 1: assert False)
- Traceability docstrings: `Validates: req-001`
- Given/When/Then as comments

**Future** (Session 2+):
```
tests/
├── unit/
│   ├── test_parser_unit.py
│   └── test_comparator_unit.py
├── integration/
│   └── test_workflow_integration.py
└── fixtures/
    ├── conftest.py
    └── sample_data/
```

## File Naming Conventions

### Python Files

**Modules**: `snake_case.py`
- `document_parser.py` (not documentParser.py)
- `comparison_engine.py` (not comparison-engine.py)

**Tests**: `test_{module_name}.py`
- `test_document_parser.py`
- `test_comparison_engine.py`

**Scripts**: `verb_noun.py`
- `build_types.py`
- `validate_all.py`

### Directories

**Modules**: `snake_case/`
- `document_parser/` (from "document-parser" or "documentParser")
- `comparison_engine/`

**Convention**: Convert kebab-case and camelCase to snake_case for Python directories

## Import Organization

### Standard Import Order

```python
# 1. Future imports
from __future__ import annotations

# 2. Standard library
import json
from pathlib import Path

# 3. Third-party
import pytest
from pydantic import BaseModel

# 4. Local - Generated types
from generated.models import Template, Section

# 5. Local - Application modules
from src.modules.document_parser import parse_template
```

### Type Hint Imports

**Use `from __future__ import annotations`** for forward references:

```python
from __future__ import annotations

from generated.models import Template

def parse_template() -> Template:  # ✅ Works with future import
    ...
```

**Without future import, need quotes**:
```python
def parse_template() -> "Template":  # ❌ Verbose
    ...
```

## Module Organization Patterns

### Pattern 1: Co-located (Session 1 Baseline)

**All module functions in one file**:

```
src/modules/document_parser/
├── __init__.py
└── parser.py              # parse_template(), parse_contract()
```

**Pros**:
- Simple structure
- Easy to review
- One file to validate

**Cons**:
- Large files if >10 functions

**Use When**: Module has ≤10 functions

### Pattern 2: Split by Function (Future)

**One file per major function**:

```
src/modules/document_parser/
├── __init__.py
├── parse_template.py      # parse_template()
└── parse_contract.py      # parse_contract()
```

**Pros**:
- Smaller files
- Clear separation

**Cons**:
- More files to manage

**Use When**: Module has >10 functions

### Pattern 3: Subdomain Organization (Future)

**Group related functions**:

```
src/modules/comparison_engine/
├── __init__.py
├── structural/
│   ├── compare_sections.py
│   └── compare_tables.py
└── content/
    └── compare_text.py
```

**Use When**: Module has >20 functions with clear subdomains

**Content Sources**:

### `ARCHITECTURE_DECISIONS.md` - Design Rationale

**Purpose**: Document key design decisions made during app-builder pipeline.

**Generated by**: Phase 8 (Completeness Documentation) using `scripts/generate_adrs.py`

**Structure** (follows ADR format):
```markdown
# Architecture Decisions Record

## ADR-{number}: {Decision Title}

**Status**: Accepted | Deferred | Rejected
**Decision Date**: YYYY-MM-DD
**Phase**: Schema | Implementation Spec | Code Generation

### Context
{Problem or requirement driving decision}

### Decision
{What was decided}

### Rationale
- {Reason 1}
- {Reason 2}

### Consequences
**Benefits**: {positive outcomes}
**Trade-offs**: {negative outcomes}
**Mitigation**: {how to address trade-offs}

### Alternatives Considered
```

**Content Sources**:
- Implementation spec ambiguity resolutions (Fix #3)
- Schema extension decisions (Fix #2)
- Code generation technology choices

**Examples**:
- ADR-001: Code-Based Extraction (No LLM)
- ADR-002: Gap Detection Deferred to Manual Implementation
- ADR-003: Session Management Models Only

**Usage**:
- Reference when making implementation decisions
- Understand why certain approaches were chosen
- Add new ADRs as design evolves
- Mark as "Superseded" when decisions change

## Best Practices

### Keep It Flat (Session 1)

**Avoid premature organization**:
- Don't create subdirectories until >10 files in a directory
- Don't split modules until >10 functions
- Don't create utils/ until code is duplicated 3+ times

### Follow Python Conventions

- `snake_case` for everything (files, directories, functions)
- `__init__.py` in every package directory
- One blank line between imports and code
- Two blank lines between top-level definitions

### Make Generated Code Obvious

**Headers in generated files**:
```python
"""
AUTO-GENERATED - DO NOT EDIT MANUALLY

Generated from: Layer 2.5 implementation specifications
Timestamp: 2025-10-31 12:34:56 UTC
...
"""
```

**Clear separation**:
- Generated code: `generated/`
- Implementation code: `src/`
- Test code: `tests/`
- Build scripts: `scripts/`

### Commit Generated Files (Controversial but Recommended)

**Why commit `generated/models.py`?**
- Drift detection: Schema changes show as model diffs in PRs
- Code review: Reviewers see type changes
- Reproducibility: Same schema → same models

**Why NOT commit?**
- Can be regenerated deterministically
- Noise in git history

**Decision**: COMMIT for Session 1 (aids validation and review)

## Anti-Patterns

### ❌ Deep Nesting Too Early

**Wrong**:
```
src/
└── core/
    └── domain/
        └── entities/
            └── models/
                └── parser/
                    └── document/
                        └── parse_template.py
```

**Right** (Session 1):
```
src/
└── modules/
    └── document_parser/
        └── parser.py
```

### ❌ Mixed Naming Conventions

**Wrong**:
```
src/modules/
├── documentParser/    # camelCase
├── comparison-engine/ # kebab-case
└── validator_service/ # snake_case
```

**Right**:
```
src/modules/
├── document_parser/   # snake_case everywhere
├── comparison_engine/
└── validator_service/
```

### ❌ Editing Generated Files

**Wrong**: Manually editing `generated/models.py`

**Right**: Edit `schema.json`, then run `python scripts/build_types.py`
