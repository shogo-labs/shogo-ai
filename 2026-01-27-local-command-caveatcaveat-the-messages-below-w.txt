  Directory Selection (getDefaultOutputDir)

  function getDefaultOutputDir(projectName: string): string {
    // Priority 1: PROJECT_DIR env var (Kubernetes runtime)
    const projectDir = process.env.PROJECT_DIR
    if (projectDir && existsSync(projectDir)) {
      return projectDir  // e.g., /app/project
    }

    // Priority 2: PROJECT_ID with workspaces directory
    const projectId = process.env.PROJECT_ID
    if (projectId) {
      const workspacesDir = resolve(MONOREPO_ROOT, "workspaces")
      if (existsSync(workspacesDir)) {
        return resolve(workspacesDir, projectId)
      }
    }

    // Priority 3: workspaces/{name} for local development
    const workspacesDir = resolve(MONOREPO_ROOT, "workspaces")
    if (existsSync(workspacesDir)) {
      return resolve(workspacesDir, projectId || projectName)
    }

    // Priority 4: Fallback to cwd
    return process.cwd()
  }

вҸә createServerFn is TanStack Start's RPC (Remote Procedure Call) mechanism. It
  lets you write server-side functions that are automatically callable from the
  client.

  How It Works

  You write this:
  // src/server/users.ts
  export const getUserList = createServerFn({ method: 'POST' })
    .inputValidator(z.object({ limit: z.number().optional() }))
    .handler(async ({ data }) => {
      // This runs on the SERVER
      const users = await prisma.user.findMany({ take: data.limit })
      return users
    })

  You call it like this:
  // src/components/UserList.tsx (CLIENT)
  import { getUserList } from '../server/users'

  function UserList() {
    const [users, setUsers] = useState([])

    useEffect(() => {
      // Looks like a function call, but it's actually an HTTP request
      getUserList({ data: { limit: 10 } }).then(setUsers)
    }, [])
  }

  Why Use This?

  - Type safety - Input/output types flow to the client automatically
  - No API routes - No need to manually create /api/users endpoints

  - Automatic serialization - Handles JSON conversion

  It's similar to tRPC, Next.js Server Actions, or Remix's loaders/actions -
  just TanStack's take on the pattern.

вқҜ so why cant react native use this 

вҸә React Native can call these endpoints - they're just HTTP requests under the
  hood. The issue is the developer experience magic that makes it look like a
  function call.

  The Magic is in the Build Step

  TanStack Start's Vite plugin transforms your code:

  // What you write (client-side)
  const users = await getUserList({ data: { limit: 10 } })

  // What Vite transforms it to (simplified)
  const users = await fetch('/_server/getUserList', {
    method: 'POST',
    body: JSON.stringify({ limit: 10 })
  }).then(r => r.json())

  React Native uses Metro bundler, not Vite, so it doesn't have this transform.

  What React Native Would Need

  Option 1: Just use fetch directly
  // React Native client
  const users = await fetch('https://your-server.com/_server/getUserList', {
    method: 'POST',
    body: JSON.stringify({ limit: 10 })
  }).then(r => r.json())

  This works, but you lose type safety and the nice API.

  Option 2: Generate a typed client
  // Auto-generated client for React Native
  export const api = {
    getUserList: (data: { limit?: number }) =>
      fetch(`${BASE_URL}/_server/getUserList`, {
        method: 'POST',
        body: JSON.stringify(data)
      }).then(r => r.json() as Promise<User[]>)
  }

  Option 3: Use tRPC or similar

  tRPC works with any client because it's not tied to a specific bundler - it
  just needs an HTTP client.

  So the Answer Is...

  It's not that React Native can't use it - it's that you'd need to either:
  1. Generate a fetch-based client alongside the server functions
  2. Use a framework-agnostic RPC solution (tRPC, Hono RPC, etc.)
  3. Have users manually call the endpoints with fetch

  The templates could include a TanStack Start backend that serves both web
  (with the magic) and mobile (via generated client or raw fetch).

вқҜ what is tRPC. be brief 

вҸә tRPC is TypeScript RPC - it lets you call backend functions from the frontend
  with full type safety, without code generation.

  // Server
  const appRouter = router({
    getUser: publicProcedure
      .input(z.object({ id: z.string() }))
      .query(({ input }) => db.user.findUnique({ where: { id: input.id } }))
  })

  // Client - types flow automatically
  const user = await trpc.getUser.query({ id: '123' })
  //    ^? User (fully typed, no codegen)

  Key points:
  - Types inferred from server to client via TypeScript (no codegen step)
  - Works with any client (React, React Native, Node, etc.) - just needs fetch
  - Framework agnostic - runs on Express, Next.js, Hono, standalone, etc.

  It's what createServerFn is inspired by, but tRPC isn't tied to a specific
  bundler.

вқҜ can you explain what prisma is exactly and what is prisma accelerate 

вҸә Prisma

  Prisma is a TypeScript ORM (Object-Relational Mapper). It lets you interact
  with databases using typed JavaScript instead of raw SQL.

  You define a schema:
  // prisma/schema.prisma
  model User {
    id    String @id @default(uuid())
    email String @unique
    todos Todo[]
  }

  model Todo {
    id     String  @id @default(uuid())

вҸә createServerFn is TanStack Start's RPC (Remote Procedure Call) mechanism. It
  lets you write server-side functions that are automatically callable from the
  client.

  How It Works

  You write this:
  // src/server/users.ts
  export const getUserList = createServerFn({ method: 'POST' })
    .inputValidator(z.object({ limit: z.number().optional() }))
    .handler(async ({ data }) => {
      // This runs on the SERVER
      const users = await prisma.user.findMany({ take: data.limit })
      return users
    })

  You call it like this:
  // src/components/UserList.tsx (CLIENT)
  import { getUserList } from '../server/users'

  function UserList() {
    const [users, setUsers] = useState([])

    useEffect(() => {
      // Looks like a function call, but it's actually an HTTP request
      getUserList({ data: { limit: 10 } }).then(setUsers)
    }, [])
  }

вқҜ so why cant react native use this 

вҸә React Native can call these endpoints - they're just HTTP requests under the
  hood. The issue is the developer experience magic that makes it look like a
  function call.

  The Magic is in the Build Step

  TanStack Start's Vite plugin transforms your code:

  // What you write (client-side)
  const users = await getUserList({ data: { limit: 10 } })

  // What Vite transforms it to (simplified)
  const users = await fetch('/_server/getUserList', {
    method: 'POST',
    body: JSON.stringify({ limit: 10 })
  }).then(r => r.json())

  React Native uses Metro bundler, not Vite, so it doesn't have this transform.

  What React Native Would Need

  Option 1: Just use fetch directly
  // React Native client
  const users = await fetch('https://your-server.com/_server/getUserList', {
    method: 'POST',
    body: JSON.stringify({ limit: 10 })
  }).then(r => r.json())

  This works, but you lose type safety and the nice API.

  Option 2: Generate a typed client
  // Auto-generated client for React Native
  export const api = {
    getUserList: (data: { limit?: number }) =>
      fetch(`${BASE_URL}/_server/getUserList`, {
        method: 'POST',
        body: JSON.stringify(data)
      }).then(r => r.json() as Promise<User[]>)
  }

  Option 3: Use tRPC or similar

  tRPC works with any client because it's not tied to a specific bundler - it
  just needs an HTTP client.

  So the Answer Is...

  It's not that React Native can't use it - it's that you'd need to either:
  1. Generate a fetch-based client alongside the server functions
  2. Use a framework-agnostic RPC solution (tRPC, Hono RPC, etc.)
  3. Have users manually call the endpoints with fetch

  The templates could include a TanStack Start backend that serves both web
  (with the magic) and mobile (via generated client or raw fetch).

вқҜ what is tRPC. be brief 

вҸә tRPC is TypeScript RPC - it lets you call backend functions from the frontend
  with full type safety, without code generation.

  // Server
  const appRouter = router({
    getUser: publicProcedure
      .input(z.object({ id: z.string() }))
      .query(({ input }) => db.user.findUnique({ where: { id: input.id } }))
  })

  // Client - types flow automatically
  const user = await trpc.getUser.query({ id: '123' })
  //    ^? User (fully typed, no codegen)

  Key points:
  - Types inferred from server to client via TypeScript (no codegen step)
  - Works with any client (React, React Native, Node, etc.) - just needs fetch
  - Framework agnostic - runs on Express, Next.js, Hono, standalone, etc.

  It's what createServerFn is inspired by, but tRPC isn't tied to a specific
  bundler.

вқҜ can you explain what prisma is exactly and what is prisma accelerate 

вҸә Prisma

  Prisma is a TypeScript ORM (Object-Relational Mapper). It lets you interact
  with databases using typed JavaScript instead of raw SQL.

  You define a schema:
  // prisma/schema.prisma
  model User {
    id    String @id @default(uuid())
    email String @unique
    todos Todo[]
  }

  model Todo {
    id     String  @id @default(uuid())
    title  String
    done   Boolean @default(false)
    user   User    @relation(fields: [userId], references: [id])
    userId String
  }

  Prisma generates a typed client:
  // Fully typed - autocomplete, error checking, etc.
  const user = await prisma.user.findUnique({
    where: { email: 'alice@example.com' },
    include: { todos: true }  // Type-safe relations
  })

  // user.todos is typed as Todo[]

  It handles:
  - Migrations (prisma migrate)
  - Schema syncing (prisma db push)
  - Query building (no raw SQL needed)
  - Connection pooling

  ---
  Prisma Accelerate

  Prisma Accelerate is a managed proxy/cache layer that sits between your app
  and database.

вҸә What prismaToRoutesCode Does

  It's a code generator that reads your Prisma schema and outputs a complete
  Hono API server with CRUD routes.

  Input: Prisma schema
  model Todo {
    id        String   @id @default(uuid())
    title     String
    done      Boolean  @default(false)
    userId    String
  }

  model User {
    id    String @id @default(uuid())
    email String @unique
    todos Todo[]
  }

  Output: Generated Hono routes
  // Auto-generated - DO NOT EDIT

  // GET    /api/todos        вҶ’ list all todos
  // GET    /api/todos/:id    вҶ’ get one todo
  // POST   /api/todos        вҶ’ create todo
  // PATCH  /api/todos/:id    вҶ’ update todo
  // DELETE /api/todos/:id    вҶ’ delete todo

  // GET    /api/users        вҶ’ list all users
  // GET    /api/users/:id    вҶ’ get one user
  // POST   /api/users        вҶ’ create user
  // PATCH  /api/users/:id    вҶ’ update user
  // DELETE /api/users/:id    вҶ’ delete user

  How It's Used

  import { prismaToRoutesCode } from "@shogo/state-api/generators"
  import { writeFileSync } from "fs"

  // Generate code from schema
  const result = await prismaToRoutesCode({
    schemaPath: "./prisma/schema.prisma",
    models: ["Todo", "User"],  // optional: filter which models
  })

  // Write to file
  writeFileSync("./src/generated/routes.ts", result.code)

  Then in your server:
  import { Hono } from "hono"
  import { createGeneratedRoutes } from "./generated/routes"
  import { PrismaClient } from "@prisma/client"

  const app = new Hono()
  const prisma = new PrismaClient()

  // Mount all generated CRUD routes at /api
  app.route("/api", createGeneratedRoutes({ prisma }))

вҸә An API client generator creates typed fetch wrappers for calling the server
  routes from the client.

  What You Have Now (Server Generator)

  prismaToRoutesCode() generates the server:

  // Generated server code (runs on Node.js)
  router.get("/todos", async (c) => {
    const items = await prisma.todo.findMany()
    return c.json({ ok: true, items })
  })

  router.post("/todos", async (c) => {
    const body = await c.req.json()
    const item = await prisma.todo.create({ data: body })
    return c.json({ ok: true, data: item })
  })

  What's Missing (Client Generator)

  A client generator would create typed fetch wrappers for the mobile app:

  // Generated client code (runs on mobile/browser)
  export const api = {
    todos: {
      list: async (params?: { limit?: number; offset?: number }) => {
        const res = await fetch(`${BASE_URL}/api/todos?${new 
  URLSearchParams(params)}`)
        return res.json() as Promise<{ ok: true; items: Todo[] }>
      },

      get: async (id: string) => {
        const res = await fetch(`${BASE_URL}/api/todos/${id}`)
        return res.json() as Promise<{ ok: true; data: Todo }>
      },

      create: async (data: { title: string; done?: boolean }) => {
        const res = await fetch(`${BASE_URL}/api/todos`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify(data),
        })
        return res.json() as Promise<{ ok: true; data: Todo }>
      },

      update: async (id: string, data: Partial<Todo>) => { ... },
      delete: async (id: string) => { ... },
    },

    users: {
      list: async () => { ... },
      get: async (id: string) => { ... },
      // ...
    },
  }

  Usage in Expo App

  // Instead of raw fetch with no types:
  const res = await fetch('/api/todos')
  const data = await res.json() // any рҹҳў

  // You get typed methods:
  const { items } = await api.todos.list()
  //      ^? Todo[] (fully typed! рҹҺү)

  const { data: todo } = await api.todos.create({ title: 'Buy milk' })

вҸә Currently the SDK generates MobX stores that use createServerFn for data
  fetching. For Expo, those same stores need to use the HTTP API client instead.

  Current: Web Template Stores

  // Generated store for TanStack Start web app
  import { getTodoList, createTodo, updateTodo, deleteTodo } from
  '../server/todos'

  class TodoStore {
    items: Todo[] = []

    async loadAll() {
      // Uses createServerFn - only works with Vite bundler
      const items = await getTodoList({ data: { userId: this.userId } })
      runInAction(() => { this.items = items })
    }

    async add(title: string) {
      // Server function call
      const todo = await createTodo({ data: { title, userId: this.userId } })
      runInAction(() => { this.items.push(todo) })
    }

    async toggle(id: string) {
      const todo = this.items.find(t => t.id === id)
      // Server function call
      await updateTodo({ data: { id, done: !todo.done } })
      runInAction(() => { todo.done = !todo.done })
    }
  }

  Expo: Same Store, Different Fetching

  // Generated store for Expo app
  import { api } from '../generated/client'

  class TodoStore {
    items: Todo[] = []

    async loadAll() {
      // Uses HTTP client - works anywhere
      const { items } = await api.todos.list({ userId: this.userId })
      runInAction(() => { this.items = items })
    }

    async add(title: string) {
      // HTTP POST
      const { data: todo } = await api.todos.create({ title, userId: this.userId
   })
      runInAction(() => { this.items.push(todo) })
    }

    async toggle(id: string) {
      const todo = this.items.find(t => t.id === id)
      // HTTP PATCH
      await api.todos.update(id, { done: !todo.done })
      runInAction(() => { todo.done = !todo.done })
    }
  }

  The Generator Flag

  Instead of maintaining two separate store generators, use a flag:

  # Generate for TanStack Start (web)
  shogo generate stores --target web

  # Generate for Expo (mobile)
  shogo generate stores --target expo

  The generator would swap out the data layer:

  // Inside the generator
  function generateStore(model: Model, target: 'web' | 'expo') {
    if (target === 'web') {
      return `
        import { get${model.name}List, create${model.name} } from 
  '../server/${model.name}'
        
        async loadAll() {
          const items = await get${model.name}List({ data: { userId } })
          // ...
        }
      `
    } else {
      return `
        import { api } from '../generated/client'
        
        async loadAll() {
          const { items } = await api.${model.nameLower}.list({ userId })
          // ...
        }
      `
    }
  }
